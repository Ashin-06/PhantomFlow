1. System Overview & Project Objectives

PhantomFlow is an enterprise-grade network threat detection platform specializing in identifying **covert communications, machine learning anomalies, and ransomware actions** in encrypted and decrypted network packet sequences. Unlike traditional Network Intrusion Detection Systems (NIDS) that inspect unencrypted headers and payloads or require expensive TLS decryption proxies, PhantomFlow analyzes encrypted traffic without decryption by tracking flow statistics, temporal frequencies, and packet sequence dimensions.

The platform bridges machine learning anomaly pipelines (e.g., LSTMs, Isolation Forests) with structural behavioral rule-based threat modules. This dual approach ensures high-performance detection across a broad attack surface, capturing C2 beacons, DNS tunnels, data exfiltration, horizontal/vertical port scans, SSH/RDP brute forcing, lateral moves, and ransomware behavior.

Project Genesis & Research Evolution

PhantomFlow transitioned from a synthetic laboratory prototype to a production-hardened threat appliance through a systematic, data-driven methodology. The engineering timeline is broken down into six developmental phases:

PHASE 1
Synthetic 1.0 (NetworkTrafficGAN)
Evaluated models against synthetic GAN network sequences. Captured perfect 100% metrics across all estimators. However, audit analysis revealed heavy shortcut learning (memorizing TCP sequence numbers and flags specific to the GAN model).
PHASE 2
Real Data Transition (CTU-13 & CICIDS)
Re-trained models against active botnet recordings. The C2 and Exfiltration models collapsed when tested under Out-of-Distribution (OOD) scenarios (e.g., training on Neris HTTP botnet, and evaluating on Murlo IRC botnet). This forced a shift away from protocol-specific metrics.
PHASE 3
Protocol Invariant Discovery & Calibration
Conducted ablation studies and removed 44 columns, including protocols, TCP flags, and ports. Retained only 6 invariant flow dimensions. Benchmarks showed recall stabilized at 79.55%, although Stage 1 precision dropped to 7.51% due to benign periodic signals (NTP/DNS).
PHASE 4
Stage 2 Precision Recovery Audit
Analyzed heuristic and supervised model additions to filter out False Positives. Early heuristic filters dropped OOD recall for Virut P2P to 37.3%. Random Forest meta-estimators overfitted to volumetric shapes. We determined that Stage 2 must rely strictly on temporal sequence-interval patterns.
PHASE 5
Temporal-Only Tuning & Exfil Validation
Ablated volumetric values from the Random Forest. Trained using only temporal metrics over a history of up to 15 flows. Hardened the Exfiltration subsystem against coordination failures. Yielded 89% precision on OOD Murlo IRC, and a 0.74 F1-score on organic exfiltration validation sets.
PHASE 6
Production Calibration & Metric Swell (Current)
Injected explicit negative benign periodic templates (NTP, DNS heartbeats) into the Stage 1 training set, doubling baseline precision to 12.84%. Integrated autocorrelation features into Stage 2, achieving 77.08% precision. Exfiltration was updated to use a destination IP context filter, yielding 100% precision.

A. Raw PCAP Capture and Sniffer Filters

The capture tier (capture/pcap_fallback.py) utilizes Python's scapy library to snif traffic. To keep management traffic from polluting the threat telemetry logs, internal loopbacks and administrator interface ports are excluded:

python# Port exclusion and loopback screening inside capture/pcap_fallback.py
def inspect_frame(pkt):
    if not pkt.haslayer(IP):
        return
    ip_src = pkt[IP].src
    ip_dst = pkt[IP].dst

    # Exclude internal loopback diagnostics
    if ip_src in ("127.0.0.1", "::1") or ip_dst in ("127.0.0.1", "::1"):
        return

    # Exclude admin and database sockets
    system_ports = {9092, 9093, 6379, 5432, 5433, 8000, 8080}
    sport = pkt[TCP].sport if pkt.haslayer(TCP) else (pkt[UDP].sport if pkt.haslayer(UDP) else 0)
    dport = pkt[TCP].dport if pkt.haslayer(TCP) else (pkt[UDP].dport if pkt.haslayer(UDP) else 0)

B. Invariant Feature Engineering

The **Feature Extractor** (features/extractor.py) converts raw packet streams into statistical representations. Removing protocol-specific columns forces the ML models to learn general patterns, making them resilient to protocol evasion:

  • IAT Coefficient of Variation (CV): Calculated as $\sigma_{\text{IAT}} / \mu_{\text{IAT}}$. Periodic C2 beacons show static intervals, meaning standard deviation is near zero ($CV \approx 0.0$). Benign user traffic is highly irregular ($CV \gg 1.0$).
  • Shannon Entropy of SNI Names: Identifies DNS tunneling and domain generation algorithms (DGA) by analyzing character distribution randomness ($H(X) \ge 4.5$).
  • Bytes Ratio: Ratio of sent-to-received bytes ($\frac{\text{orig\_bytes}}{\text{resp\_bytes} + 1}$), which helps isolate exfiltration events.

7. Human-in-the-Loop Active Response

The active response suite supports IP blocking, DNS sinkholing, and EDR agent isolation. To prevent accidental outages of critical services, the platform enforces strict approval policies:

Suppression Rules Engine

Suppression rules allow analysts to filter out known, benign activities (like administrative network scans or scheduled transfers). If a network flow matches an active suppression rule, the alert is silenced: it is stored in the database but will not trigger visual banners or populate metric counters.

When a new suppression rule is created, the system also runs a **retroactive suppression check** in api/routes/suppression.py. This updates matching, unreviewed alerts in the database to a status of 'suppressed'. This ensures that the dashboard instantly cleans up related alerts without requiring a page reload.

3

Identify Verdict & Retrain

Determine whether the alert is malicious (True Positive) or benign (False Positive):

9. Operations & Database

A. PostgreSQL Database Schema

The platform database schema (PostgreSQL 15) handles persistent storage for flows, alerts, response audits, and suppression rules:

SQL-- Schema layout for the PhantomFlow database engine

-- 1. Ingested Flow Records
CREATE TABLE flows (
    flow_id           UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    timestamp         TIMESTAMPTZ NOT NULL,
    src_ip            INET NOT NULL,
    dst_ip            INET NOT NULL,
    src_port          INTEGER,
    dst_port          INTEGER,
    protocol          VARCHAR(10),
    sni               TEXT,
    orig_bytes        BIGINT,
    resp_bytes        BIGINT,
    bytes_ratio       FLOAT,
    iat_mean_ms       FLOAT,
    iat_cv            FLOAT,
    periodicity_score FLOAT,
    dns_query         TEXT,
    dns_entropy       FLOAT,
    created_at        TIMESTAMPTZ DEFAULT NOW()
);

-- 2. Security Alerts Table
CREATE TABLE alerts (
    alert_id          UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    flow_id           UUID REFERENCES flows(flow_id) ON DELETE CASCADE,
    timestamp         TIMESTAMPTZ NOT NULL,
    threat_type       VARCHAR(50) NOT NULL,
    severity          VARCHAR(20) NOT NULL,
    confidence        FLOAT NOT NULL,
    shap_values       JSONB,
    explanation       TEXT,
    mitre_ttps        TEXT[],
    analyst_status    VARCHAR(20) DEFAULT 'unreviewed' 
                      CHECK (analyst_status IN ('unreviewed', 'confirmed_tp', 'false_positive', 'escalated', 'suppressed')),
    analyst_id        VARCHAR(100),
    analyst_notes     TEXT,
    reviewed_at       TIMESTAMPTZ,
    created_at        TIMESTAMPTZ DEFAULT NOW()
);

-- 3. Active Response Audit Logs
CREATE TABLE response_audit (
    audit_id          UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    alert_id          UUID REFERENCES alerts(alert_id) ON DELETE CASCADE,
    action            VARCHAR(50) NOT NULL,
    target_ip         INET,
    target_domain     TEXT,
    approved_by       VARCHAR(100), -- "SYSTEM_AUTO" or analyst username
    approved_at       TIMESTAMPTZ,
    expires_at        TIMESTAMPTZ NOT NULL,
    executed_at       TIMESTAMPTZ,
    status            VARCHAR(20) DEFAULT 'pending' 
                      CHECK (status IN ('pending', 'executed', 'revoked', 'expired', 'failed')),
    error_detail      TEXT,
    created_at        TIMESTAMPTZ DEFAULT NOW()
);

-- 4. Suppression Rules Registry

B. Reset Suite Mechanics

To support testing, developer controls expose a **Reset Suite** action in the header. Triggering this button calls the backend route /api/alerts/reset. The endpoint performs two operations:

  1. Runs database truncations on Postgres tables (deleting all flows, alerts, response audit history, and custom suppression rules).
  2. Flushes the Redis cache (wiping all HyperLogLog tracking structures, brute-force counters, and active metrics).

This returns the system to a clean, fresh state for testing and validation.
