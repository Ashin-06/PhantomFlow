1. System Overview & Project Objectives

PhantomFlow is an enterprise-grade network threat detection platform specializing in identifying **covert communications, machine learning anomalies, and ransomware actions** in encrypted and decrypted network packet sequences. Unlike traditional Network Intrusion Detection Systems (NIDS) that inspect unencrypted headers and payloads or require expensive TLS decryption proxies, PhantomFlow analyzes encrypted traffic without decryption by tracking flow statistics, temporal frequencies, and packet sequence dimensions.

The platform bridges machine learning anomaly pipelines (e.g., LSTMs, Isolation Forests) with structural behavioral rule-based threat modules. This dual approach ensures high-performance detection across a broad attack surface, capturing C2 beacons, DNS tunnels, data exfiltration, horizontal/vertical port scans, SSH/RDP brute forcing, lateral moves, and ransomware behavior.

Project Genesis, Evaluation, and Tuning Phases

PhantomFlow transitioned from a synthetic laboratory prototype to a production-hardened threat appliance through a systematic, data-driven methodology. The engineering timeline is broken down into six developmental phases:

1

Synthetic Phase 1.0 (NetworkTrafficGAN)

Initial models were evaluated using synthetic datasets generated via Generative Adversarial Networks (GANs). The classifiers returned perfect (1.00) F1-scores. Code audits revealed the models were memorizing GAN-specific artifacts (like static TCP flags, specific packet numbers) rather than learning generalized threat structures. This phenomenon is known as "shortcut learning."

2

Real-World Dataset Transition (CTU-13 & CICIDS)

To break shortcut learning, the pipeline was transitioned to raw, real-world botnet recordings (CTU-13, containing active Neris, Murlo, and Virut families). Models trained on one botnet protocol (e.g., Neris HTTP) completely collapsed when evaluated against another (e.g., Murlo IRC), highlighting a failure to generalize under Out-of-Distribution (OOD) conditions.

3

Protocol-Invariant Calibration

We conducted ablation studies and discarded 44 fields (including protocol numbers, transport flags, and port metadata) from the core model inputs. This left only 6 invariant volumetric features (durations, bytes ratios, packet distributions). Removing specific protocols forced the model's decision boundary to learn structural traffic constraints. Under this OOD benchmark, model recall stabilized at 79.55%, though Stage 1 precision remained low at 7.51%.

4

Stage 2 Precision Recovery Audit

A precision recovery audit was launched to filter false alarms out of Stage 1. Early heuristic approaches dramatically hurt recall (dropping it to 37.3%). Supervised Random Forests trained on volumetric dimensions overfitted to the training protocols. The engineering group decided that Stage 2 must rely entirely on temporal and sequence-frequency patterns to maintain OOD recall.

5

Temporal-Only Context Engineering & Exfiltration Hardening

Stage 2 was updated to use a temporal-only feature subset (coefficient of variation of inter-arrival times, sequence trends, duration variances) over a history of up to 15 flows. In tandem, the Exfiltration subsystem was updated to handle missing features (NaNs) and tested against unseen organic streams. This raised OOD C2 precision to 89%, and stabilized exfiltration detection at a 0.74 F1-score.

6

Production Calibration & Evaluation (Current State)

To refine the appliance for production, we injected explicit negative benign periodic templates (like NTP queries, DNS health checks, AV updates) directly into the training loop, raising Stage 1 baseline precision to 12.84%. The Stage 2 Random Forest was calibrated at a decision threshold of 0.45, achieving **77.08% precision** on OOD protocols. The Exfiltration detector was hardened using destination-IP history context, yielding **100% precision** with zero false alerts on benign volumetric spikes.

2. Data Ingestion & Feature Extraction

A. Raw PCAP Capture and Sniffer Filters

The ingestion architecture starts at the Packet Sniffer (capture/pcap_fallback.py). This sniffer attaches to the primary network interface via Python's scapy library. To prevent infinite loops and internal communication spikes from polluting threat telemetry database structures, loopback packets and platform system ports are filtered out at the kernel/socket wrapper layer:

python# Ingestion filtering inside capture/pcap_fallback.py
def should_filter_packet(src_ip, dst_ip, sport, dport):
    # Rule 1: Exclude internal system loopback loop and interface communications
    if src_ip in ("127.0.0.1", "::1") or dst_ip in ("127.0.0.1", "::1"):
        return True
        
    # Rule 2: Exclude infrastructure and microservices management ports
    # 9092/9093 (Kafka), 6379 (Redis), 5432/5433 (Postgres), 8000 (FastAPI API), 8080 (UI Server)
    system_ports = {9092, 9093, 6379, 5432, 5433, 8000, 8080}
    if sport in system_ports or dport in system_ports:
        return True
        
    return False

Packets passing these validation gates are accumulated into 60-second flow objects. Once a window closes, flow objects are serialized to JSON and published to the network-flows Kafka topic running on port 9092.

B. Feature Engineering Equations

When the Kafka consumer in pipeline/orchestrator.py pulls a raw flow session, the Feature Extractor (features/extractor.py) maps packet sequences into statistical, protocol-invariant vectors. The key equations are defined below:

📈 Inter-Arrival Time Coefficient of Variation (IAT CV)

This metric represents interval frequency variance divided by mean interval size. In periodic command-and-control beacon structures, the interval is static, meaning IAT variance approaches zero, leading to IAT CV ≈ 0. Human web browsing yields highly irregular intervals, making IAT CV >> 1:

\mu = \frac{1}{N}\sum_{i=1}^{N} IAT_i, \quad \sigma = \sqrt{\frac{1}{N}\sum_{i=1}^{N}(IAT_i - \mu)^2}, \quad IAT\ CV = \frac{\sigma}{\mu}

🌐 Shannon Entropy of SNI / DNS Names

This metric measures domain label character randomness. Attackers tunneling data over DNS, or using domain generation algorithms (DGA) to avoid static list blockages, produce high-entropy strings:

H(X) = -\sum_{i=1}^{M} P(x_i) \log_2 P(x_i)

Where P(x_i) is the frequency of occurrence of character x_i in the target server name indication string or domain string of length M.

📊 Volumetric Invariant Ratios

These track physical data movement dynamics, isolating exfiltration events (high source payload, low destination response payload) regardless of the application protocol:

Bytes\ Ratio = \frac{\text{orig\_bytes}}{\text{resp\_bytes} + 1}, \quad Packets\ Rate = \frac{\text{orig\_pkts}}{\text{duration\_s}}

3. Machine Learning Model Architectures

A. Two-Stage Decoupled Detection Engine

PhantomFlow runs a two-stage classification strategy to optimize general threat detection under out-of-distribution (OOD) scenarios. This approach captures malware families that use unknown custom protocols:

Architecture Phase Model Type Inputs & Dimensions Primary Objective
Stage 1 (Invariant Detection) SGD Logist. Regression (Calibrated) 6 Invariant Columns: duration_s, total_bytes, orig_bytes, orig_pkts, resp_bytes, bytes_ratio Filters out obvious benign flows across the network. Captures any potential beacon or exfiltration signature broadly. High recall, low precision (7.51% precision).
Stage 2 (Precision Recovery) Random Forest Classifier Sequence Metrics (15 connection history): iat_cv, periodicity_score, dominant_period_ms, trend, duration_cv, history_len Triggered only when Stage 1 flags a flow and connection history contains at least 2 flows. Filters out false alarms using temporal periodicity features. Achieves **77.08% OOD C2 Precision**.
Exfiltration Subsystem Hybrid (Isolation Forest + XGBoost) 15 Features (Volume, Entropy, Domain size, DNS names) + Destination History Detects data exfiltration by looking for sustained upload anomalies to new or unique external IPs. Tracking target host history yields **100% Exfil Precision**.
💡 Why Decoupled Two-Stage?

Classifiers that use specific port details or protocols easily fail when threat actors shift from standard HTTP C2 protocols to custom encrypted protocols (like IRC or P2P). Using 6 invariant features in Stage 1 ensures that the network boundary is monitored regardless of the protocol. Stage 2 then uses temporal periodicity to confirm automation patterns without referencing specific protocols.

B. Rule-Based Threat Detectors

Alongside deep ML pipelines, rule-based modules run to identify structural attacks on the network. These modules track states efficiently using Redis:

🔍 Port Scan / Reconnaissance Detector (T1046, T1595)

Calculates horizontal and vertical port scans in a 60-second window. Memory overhead is minimized by tracking unique destinations in Redis HyperLogLog structures, which offer O(1) performance:

  • Vertical Scan: Triggered when a single source IP contacts >= 15 unique destination ports on a single target IP.
  • Horizontal Scan: Triggered when a single source IP contacts >= 20 unique destination hosts on a single destination port.
  • Stealth Flag: Appended when average packet payload size drops below 80 bytes (indicating empty TCP SYN scans).
↔️ Lateral Movement Detector (T1021, T1570)

Monitors internal, East-West network flow distributions. If a host contacts multiple distinct internal IPs over administrator administration ports within 5 minutes, an alert is raised:

  • SMB Spread (Port 445): Triggered if a host contacts > 4 unique internal hosts (often indicates ransomware propagation).
  • RDP Spread (Port 3389): Triggered if a host contacts > 3 unique internal targets.
  • RPC/WMI Spread (Port 135): Triggered if a host contacts > 4 unique internal hosts.
  • WinRM Spread (Port 5985): Triggered if a host contacts > 3 unique internal hosts.
  • SSH Spread (Port 22): Triggered if a host contacts > 5 unique internal hosts.
🔨 Brute Force credential Attack Detector (T1110)

Tracks failed authorization frequencies on authentication ports using Redis counters. It separates targeted brute-forcing from password spraying:

  • SSH/RDP Brute Force: Triggered if a source IP makes >= 10 SSH attempts/min, or >= 8 RDP attempts/min to a single destination.
  • Password Spray: Triggered if a source IP attempts auth connections to >= 4 unique destination hosts at >= 3 attempts/min on the same port.
☠️ Ransomware Behaviour Detector (T1486, T1490)

Correlates multiple signals within a 5-minute window to identify active ransomware. To minimize false alarms, the detector requires **at least 2 matching signals** to trigger:

  1. smb_write_burst: Large, high-frequency data write operations over Port 445 (SMB) exceeding 500KB.
  2. shadow_copy_kill_pattern: Internal host scanning/admin connections combined with RPC calls.
  3. pre_encrypt_c2: Active outbound C2 beacons detected on the same source IP.
  4. smb_enumeration: High-frequency, low-byte connections on Port 445, indicating directory mapping.

C. Analyst Feedback & Active Learning Loop

PhantomFlow adapts to its deployment network using supervised online learning. When a security analyst reviews an alert and flags it as a True Positive (TP) or False Positive (FP), the backend triggers the feedback loop (pipeline/feedback_trainer.py):

python# Ingestion active learning step inside feedback_trainer.py
def update(self, features: dict, label: int) -> bool:
    # label = 1 (TP, threat), label = 0 (FP, benign)
    if self._model is None or not hasattr(self._model, 'partial_fit'):
        return False
        
    with feedback_lock:
        X = self._features_to_vector(features)
        
        # Standardize features using the base pipeline scaler
        if self._imputer:
            X = self._imputer.transform(X)
        if self._scaler:
            X = self._scaler.transform(X)
            
        # Incremental update to the SGD model's linear weights
        self._model.partial_fit(X, [label], classes=[0, 1])
        
        # Persist feedback weights to disk
        joblib.dump({"model": self._model, "scaler": self._scaler}, "models/invariant_feedback.pkl")
        return True

This allows the model to learn and adapt to local network behaviors (like regular offsite backups or database synching) without requiring full pipeline retraining.

4. Human-in-the-Loop Active Response

PhantomFlow provides active response capabilities to contain threats. To prevent accidental lockouts of critical internal infrastructure, the system enforces a strict Human-in-the-Loop (HITL) approval policy:

Mitigation Type API Command Database Schema Mapping Automation Policy
Firewall IP Block block_ip Pushed to the target_ip column (Postgres INET type). Auto-executes only if model confidence >= 90%. Otherwise, queues in the dashboard for manual approval.
DNS Domain Sinkhole sinkhole_domain Pushed to the target_domain column (Postgres TEXT type). Auto-executes only if model confidence >= 95%. Otherwise, queues in the dashboard.
EDR Host Isolation isolate_host Pushed to the target_ip column (Postgres INET type). Never Auto-Executes. Always requires manual analyst approval and notes in the dashboard.
⚠️ Database Type Constraint Safety

In early versions, saving a domain name into the target_ip column caused Postgres to throw invalid input syntax for type inet errors. The API was updated to separate target destinations into target_ip (which validates CIDR formats) and target_domain (which accepts domain strings).

Suppression Rules Engine

Suppression rules allow analysts to filter out known, benign activities (like administrative network scans or scheduled transfers). If a network flow matches an active suppression rule, the alert is silenced: it is stored in the database but will not trigger visual banners or populate metric counters.

When a new suppression rule is created, the system also runs a **retroactive suppression check** in api/routes/suppression.py. This updates matching, unreviewed alerts in the database to a status of 'suppressed'. This ensures that the dashboard instantly cleans up related alerts without requiring a page reload.

6. Operations & Database

A. PostgreSQL Database Schema

The platform database schema (PostgreSQL 15) handles persistent storage for flows, alerts, response audits, and suppression rules:

SQL-- Schema layout for the PhantomFlow database engine

-- 1. Ingested Flow Records
CREATE TABLE flows (
    flow_id           UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    timestamp         TIMESTAMPTZ NOT NULL,
    src_ip            INET NOT NULL,
    dst_ip            INET NOT NULL,
    src_port          INTEGER,
    dst_port          INTEGER,
    protocol          VARCHAR(10),
    sni               TEXT,
    orig_bytes        BIGINT,
    resp_bytes        BIGINT,
    bytes_ratio       FLOAT,
    iat_mean_ms       FLOAT,
    iat_cv            FLOAT,
    periodicity_score FLOAT,
    dns_query         TEXT,
    dns_entropy       FLOAT,
    created_at        TIMESTAMPTZ DEFAULT NOW()
);

-- 2. Security Alerts Table
CREATE TABLE alerts (
    alert_id          UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    flow_id           UUID REFERENCES flows(flow_id) ON DELETE CASCADE,
    timestamp         TIMESTAMPTZ NOT NULL,
    threat_type       VARCHAR(50) NOT NULL,
    severity          VARCHAR(20) NOT NULL,
    confidence        FLOAT NOT NULL,
    shap_values       JSONB,
    explanation       TEXT,
    mitre_ttps        TEXT[],
    analyst_status    VARCHAR(20) DEFAULT 'unreviewed' 
                      CHECK (analyst_status IN ('unreviewed', 'confirmed_tp', 'false_positive', 'escalated', 'suppressed')),
    analyst_id        VARCHAR(100),
    analyst_notes     TEXT,
    reviewed_at       TIMESTAMPTZ,
    created_at        TIMESTAMPTZ DEFAULT NOW()
);

-- 3. Active Response Audit Logs
CREATE TABLE response_audit (
    audit_id          UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    alert_id          UUID REFERENCES alerts(alert_id) ON DELETE CASCADE,
    action            VARCHAR(50) NOT NULL,
    target_ip         INET,
    target_domain     TEXT,
    approved_by       VARCHAR(100), -- "SYSTEM_AUTO" or analyst username
    approved_at       TIMESTAMPTZ,
    expires_at        TIMESTAMPTZ NOT NULL,
    executed_at       TIMESTAMPTZ,
    status            VARCHAR(20) DEFAULT 'pending' 
                      CHECK (status IN ('pending', 'executed', 'revoked', 'expired', 'failed')),
    error_detail      TEXT,
    created_at        TIMESTAMPTZ DEFAULT NOW()
);

-- 4. Suppression Rules Registry
CREATE TABLE suppression_rules (
    rule_id           UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    name              VARCHAR(200) NOT NULL,
    description       TEXT,
    src_ip_cidr       CIDR,
    dst_ip_cidr       CIDR,
    dst_port          INTEGER,
    sni_pattern       TEXT,
    threat_type       VARCHAR(50),
    created_by        VARCHAR(100) NOT NULL,
    created_at        TIMESTAMPTZ DEFAULT NOW(),
    expires_at        TIMESTAMPTZ,
    is_active         BOOLEAN DEFAULT TRUE,
    justification     TEXT NOT NULL,
    ticket_reference  VARCHAR(50)
);

-- Index maps optimized for real-time dashboard updates
CREATE INDEX idx_alerts_timestamp ON alerts(timestamp DESC);
CREATE INDEX idx_alerts_status    ON alerts(analyst_status);
CREATE INDEX idx_flows_src_ip     ON flows(src_ip);
CREATE INDEX idx_flows_dst_ip     ON flows(dst_ip);

B. Reset Suite Mechanics

To support testing, developer controls expose a **Reset Suite** action in the header. Triggering this button calls the backend route /api/alerts/reset. The endpoint performs two operations:

  1. Runs database truncations on Postgres tables (deleting all flows, alerts, response audit history, and custom suppression rules).
  2. Flushes the Redis cache (wiping all HyperLogLog tracking structures, brute-force counters, and active metrics).

This returns the system to a clean, fresh state for testing and validation.
